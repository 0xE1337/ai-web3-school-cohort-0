## 关键收获 1：三层抽象框架（confirm 按钮背后的三问）

Bruce 把整个 Web3 流程拆成回答三个问题：

| 问题 | 涉及 |
|------|------|
| **你是谁** | 私钥 / 助记词 / 地址 / 签名 |
| **你要做什么** | 交易意图 / Gas / Nonce / 合约调用 |
| **为什么大家相信结果** | 节点传播 / Mempool 排队 / Builder 排序 / Validator 出块 / Finality |

这套框架特别清晰——之前我对 Gas、Nonce、签名是分开理解的，但 Bruce 把它们串成「**身份 → 授权 → 传播 → 排序 → 执行 → 确认**」单线流程后，整个交易生命周期立刻直观了。

**衍生细节**：
- 签名验证用 **EC recover**：把签名 + 原文 → 反推签名地址 → 跟发送方地址比对
- **量子计算威胁**：如果哈希/签名算法被攻破，攻击者可以碰撞出同一签名，"瞬间归零"风险

## 关键收获 2：Web3 = 密码学 + 经济学 + 社会学 三学科交叉

这是整场最高密度的核心观点：

| 学科 | 解决的问题 | 在 Web3 里的体现 |
|------|------------|------------------|
| **密码学** | 谁能证明你有权操作 | 签名 / 哈希 / 抗量子算法 |
| **经济学** | 谁承担成本、谁获得激励 | Gas / PoS 质押 / 罚没 / 通缩 |
| **社会学** | 规则由谁说了算 | 治理 / 共识 / 硬分叉投票 |

> "任何一环失衡，整套就跑不起来。密码学被攻破直接完蛋；经济学设计不闭环没人维护节点也不干了；失去去中心化整个网络就跟普通数据库没区别。" —— Bruce

## 关键收获 3：智能合约升级 ≠ 网络升级，是两个层级

这是 Q&A 环节被一位学员问出来的精彩澄清。很多人会把这两件事混在一起想：

### 合约层升级
- 合约代码部署后**字节码本身不可变**
- 升级靠 **proxy 代理模式**：用户调用的是一个 proxy 合约，proxy 里存了一个指针指向真正的实现合约
- 升级 = 改 proxy 里的指针，指向新的实现合约
- OZ 的几种 proxy 模式：**Transparent Proxy / UUPS / Beacon**，各有 Gas 开销和升级权限差异
- 严格说"智能合约可变"是不准确的——**字节码不可变，但执行入口可以转向**

### 网络层升级（以太坊主网）
- 升级是**硬分叉**：所有节点客户端在某个 block height 切到新规则
- 流程：Ethereum Magicians 论坛讨论 → All Core Devs 会议 → EIP 文档定稿 → 客户端实现 → 测试网验证 → 主网 feature flag 开启
- "软切换"靠生态共识：老节点没人维护就自然没落
- 历史上确实有客户端实现 bug 导致小范围分叉，但因为客户端多样性（5-6 个语言实现），网络整体没崩

Bruce 用"指针"比喻把这两层拆得很清楚，对没做过协议层的人特别友好。

## 关键收获 4：DeFi 协议升级的 7 天 timelock

成熟 DeFi 协议合约升级**不会立即生效**，要走 7 天时间锁。

### 为什么要 7 天

- 给用户预留**撤资窗口**——如果升级有问题、社区有质疑，大家有时间反应
- 给媒体/审计者**报道窗口**——升级一旦发起，所有人都能看到，自动触发社区监督
- 是一道**事中防御**，不是事后兜底

### 历史教训

> "之前有黑客是在圣诞节大家放假的时候去攻击，偷偷改了一个东西。因为有时间锁的保护，有人偷偷的发现这个合约怎么改了，紧急发现是有这个攻击，赶紧在那个生效之前给它干掉了。" —— Bruce

这个机制对**任何需要升级权限**的协议设计都是参考——把"时间"当成防御资源，比单纯靠多签更稳。
